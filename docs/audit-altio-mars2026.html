🔒 Sécurité — 6 critiques

1. getProviders() expose TOUS les champs utilisateur
src/api/profile.ts — ligne 77
user:users!inner(*) renvoie stripe_account_id, google_calendar_token, siren, billing_address… à n'importe quel utilisateur authentifié. Fix : sélectionner explicitement id, name, picture, email.
2. getProperties() et getReservations() sans filtre owner_id
src/api/properties.ts — lignes 10-17, 88-95
N'importe quel utilisateur peut lister TOUTES les propriétés et réservations de la plateforme (noms de locataires, codes d'accès, dates). Fix : ajouter .eq('owner_id', session.user.id).
3. Fonctions admin sans vérification is_admin
src/api/admin.ts — lignes 64-100
suspendUser(), rejectProviderDocument(), getAdminStats() ne vérifient pas que l'utilisateur est admin. Fix : vérifier session.user.is_admin en début de chaque fonction.
4. RLS "read all profiles" trop permissive
migration 20260321000006 — lignes 17-19
Policy USING (true) sur la table users permet à tout authentifié de lire tous les profils (SIREN, TVA, etc.). Fix : restreindre aux champs publics ou au propre profil.
5. Partner CRUD sans authentification
src/api/partners.ts — lignes 19-66
createPartner(), updatePartner(), deletePartner() n'ont aucun check auth. N'importe qui peut manipuler les partenaires. Fix : ajouter vérification admin.
6. Service role key stockée en config DB
migration 20260318000003 — lignes 29, 48-50
Le service_role_key est accessible via current_setting('app.service_role_key') depuis n'importe quelle fonction PostgreSQL. Escalade de privilèges possible.
+ 7 vulnérabilités High/Medium
7. Race condition — acceptEmergencyBid()
src/api/emergencies.ts — lignes 339-382
Deux bids peuvent être acceptés simultanément. Fix : ajouter WHERE status = 'pending' sur l'update.
8. cancelMission() sans contrainte de statut
src/api/missions.ts — lignes 660-664
L'update ne filtre pas sur le statut actuel — une mission completed ou paid peut être annulée. Fix : ajouter .in('status', ['pending', 'assigned']).
9. checkPaymentStatus() retourne toujours 'paid'
src/api/payments.ts — ligne 87
Fonction stub qui ne vérifie jamais Stripe. Risque de faux positifs sur les paiements.
10. Calcul montant imprécis (floating point)
src/api/emergencies.ts — lignes 210-211
baseCost * 1.10 * 100 peut produire des centimes incorrects. Fix : Math.round(baseCost * 110).
11. Notifications push échouent silencieusement en prod
src/api/notifications.ts — lignes 32-44
Erreurs RPC + Edge Function catchées sans logging Sentry. Aucun feedback utilisateur.
12. String interpolation dans filtre admin
src/api/admin.ts — ligne 27
or(`name.ilike.%${search}%`) — injection PostgREST théoriquement possible.
13. Email confirmation désactivée
supabase/config.toml
Les utilisateurs peuvent s'inscrire avec n'importe quelle email sans vérification.

💡 Conclusion

Le projet Altio est solide sur le plan architectural : conventions cohérentes, stack moderne (React 19, RN 0.81, Expo 54), TypeScript strict, design system centralisé, et logique métier complexe bien structurée (state machines, 18 Edge Functions, Factur-X prêt).

Les progrès récents sont significatifs : annulation d'urgences, avis/favoris, cascade deletes, notifications server-side, planning provider avec statuts. Le sprint 1 avance bien.

Le point bloquant principal est la sécurité : 6 vulnérabilités critiques (exposition de données, fonctions admin non protégées, RLS trop permissive) doivent être corrigées avant toute mise en production. La bonne nouvelle : ce sont des fixes rapides (ajout de filtres et checks auth), pas des refactos architecturaux.

Priorité immédiate : terminer Sprint 1 (bugs status) + insérer un Sprint 1.5 sécurité avant de continuer vers RGPD et facturation.
